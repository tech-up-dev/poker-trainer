Technical Specification

Poker Training & Quiz App V3

Subscription-gated, mobile-first PWA. Greenfield build with a no-code Content Operations backbone, guided Skills Path, and full engagement loop. This document is for the engineering team and excludes commercial terms.

Client: Steve Pavilanis, Bushwood Digital Inc.   ·   Engineering reference (June 4 PRD baseline)

Context

Project Context

A web-based poker training and quiz app for Bushwood Digital's poker coaching membership community. A Duolingo-style, flashcard-style learning experience built around the client's poker strategy methodology. Not a real-money platform. Not a hand-simulation engine. Content-driven.

Core loop: short lesson or concept intro, scenario-based quiz questions, instant feedback with teaching commentary, points, scoring, daily streaks, and a guided skills path. Standalone subscription product, separate from the client's existing membership platform. Delivered as a PWA so members can install it on their phones without app store involvement and receive push reminders.

Two workstreams: the member-facing app and the no-code Content Operations backbone (CMS). The CMS is the heaviest piece by design; the client has called it the heart of the app. References elsewhere to "V2" refer to additional products (Masterclass, AI features) that V1's architecture is designed to accept additively.

Locked In

Confirmed Approaches

Decisions captured from the June 4 PRD and the alignment call. These are the technical baseline for the build.

✓

GoHighLevel integration

Standard Stripe-backed GHL integration confirmed. Real-time per-Stripe-event webhook from a Supabase Edge Function pushing to GoHighLevel. No native GHL processing in scope, no additional webhook handler needed.

✓

Single content pipeline (content-type-agnostic)

The validate, stage, publish, rollback pipeline serves all V1 content types (Lessons, Tips, References, Path nodes) through the same Edge Functions and the same versioning machinery. No bespoke per-type pipeline anywhere in V1.

✓

Entitlements over subscription status

A user has zero or more entitlements, each mapped to a Stripe product / price. The app checks entitlements, not raw subscription state. Flexible price points map to a single binary "app access" entitlement, so admin can add or change prices in Stripe or GHL anytime with no code changes.

✓

Auth model

Supabase Auth: password (immediate post-purchase login, no email wait) plus magic-link as a secondary / recovery path. Single shared identity from day one.

✓

Two-layer rollback

Code rollback via Vercel's one-click deployment revert. Content rollback via Supabase versioned publish. Code and content can roll back independently without coordinating each other.

✓

Avatars client-supplied

Six player-type illustrations and brand assets provided by client or their designer. Avatar illustration is NOT part of dev hours. Integration of supplied artwork is in scope.

Build

Scope of Work

Two workstreams. The member-facing app and the Content Operations backbone. The CMS carries the heaviest technical weight.

⊞
Static 9-Max poker table & card display (member app)
  • Custom static 9-max (full-ring, nine-seat) poker table layout. NOT 6-max. Hard requirement
  • Displays positions, stack sizes, dealer button, blinds, pot, community cards, hero hole cards
  • Lightweight card-rendering component (no dealing animation, no card play)
  • Player-type avatars at relevant seats as tappable markers that open a short explainer for that player type
  • All nine seats fit cleanly on a mobile browser screen

Note on approach: Existing React poker libraries are built for animated card dealing, which this project does not need. Custom static layout reusing a lightweight card component.

⊟
Quiz & lesson engine
  • Question sequencing, per-answer scoring, session completion, final session score
  • Lesson flow: concept intro, scenario question set, instant feedback, session score
  • Scenario-based questions referencing the static table (position vs player type vs board)
  • Multiple-choice answer interface, four answers, one correct per question, feedback shown regardless of choice
  • Review missed questions at end of session
  • Starred / saved questions, persistent across sessions (refinement of missed-question review)
  • Sessions can run in randomized order (methodology is all-encompassing, no fixed prerequisite ordering)
▤
Glossary / tap-to-define (with nested linking)
  • Tappable poker terms inside questions and explanations open a definition drawer
  • Drawer shows: term, plain-language definition, importance indicator, specific-situation example, example usage
  • Nested linking: definitions can link to other definitions, with a navigation stack (back through the chain)
  • One managed glossary table serves both the in-drill tap-to-define drawer (M1) and the browsable References Library view (M2). Single source, two UI surfaces
  • Glossary is a managed library the client can edit via the Content Operations backbone
▦
Content Operations backbone (CMS): the heart of the app
  • No-code wizard-style authoring UI that branches by question type (general Q&A or hand scenario) for one-off authoring
  • Visual interactive table builder for hand scenarios: place players on a 9-max layout, assign player types, set stack sizes, move the dealer button, set hole cards and board cards. Distinct interactive editor (separate from the static table in the member app)
  • Bulk import / export as the primary content workflow: the workhorse for thousands of questions. CSV, JSON, and Markdown
  • The import schema is cleanly documented as a markdown spec the client can paste into Claude or other AI tools to generate matching output
  • Validation engine using Zod with field-path-specific, human-readable error messages so the client can fix issues without developer involvement (e.g. "question 3, answers: expected exactly one with is_correct=true, found 0")
  • Separate Supabase projects for staging and production: isolated environments from day one
  • Write-to-staging flow with preview on staging
  • Versioned publish to production: each promote creates a new content version with full history
  • One-click rollback to any previous content version, controlled by the client, no developer needed
  • Single shared scenario data model across the wizard, bulk upload, validator, and member app so all paths stay aligned

Note on approach: The CMS table builder is a custom interactive editor (drag / click to seat players, set stacks via modal, move the dealer button, assign cards from a picker). Built as its own component, not a library, because no existing React poker library targets interactive authoring of this shape.

💡
Today's Tip, Saved Tips & References Library
  • Today's Tip surface on the home / dashboard, rotating from the client's tip library
  • Members can save tips for later, organized in a personal Saved Tips list
  • References Library: cheat sheets, Character Mapping reference, methodology references
  • All authored and versioned through the CMS, same publish flow as questions and lessons
⊿
Skills Path & guided progression
  • Duolingo-style visual progression map with skill nodes and connecting paths
  • Unlock states (locked, available, in progress, mastered) driven by lesson completion and accuracy
  • Dependencies between skill nodes so the member sees a clear next step
  • Maintains the randomized-order option for free practice, while offering this guided arc on top
★
Onboarding & warm-up lesson
  • Short onboarding flow on first login, framing what the app is and how scoring / streaks work
  • Warm-up lesson: a curated low-friction first session so members get an early win
  • Celebratory animations on first-correct, first-session-complete, first-streak, first-badge
📱
Progressive Web App (PWA) & Web Push
  • PWA built in from day one: manifest, service worker, install prompt, app-like full-screen experience on Android and iOS
  • Add-to-Home-Screen support so members get an icon on their phone without app store involvement
  • Service-worker caching strategy designed in coordination with the content-publish flow so "promote to production" always reaches members cleanly and never leaves them on stale content
  • Web push notifications (VAPID setup, scheduled send job) for streak reminders
  • All screens fully mobile-responsive. Mobile browser is the priority. Most members will be on phones

Honest tradeoff: iOS PWA capabilities are slightly more limited than Android (mainly around push permission timing and some background behavior), but for a subscription quiz app this is rarely felt. Web Push is supported on iOS 16.4+ for installed PWAs.

🏆
Gamification, streaks & badge mastery
  • Points and scoring based on quiz accuracy
  • Daily streak with progress tracking and reminder push notifications
  • Monthly contest leaderboard: ranked by quiz performance, monthly reset, contest mechanics for community engagement
  • Achievement badges for milestones
  • Badge decay / mastery-maintenance: badges fade if not exercised, restored on re-engagement (keeps the loop active)
📈
Progress tracking & dashboard
  • Per-member progress: questions answered, accuracy rate, sessions completed, improvement over time
  • Dashboard showing rating, selected stats, trend over time
  • Today's Tip and outbound links to client website for upgrades and discounts
🔒
Authentication, entitlements & access (Supabase)
  • Fresh Supabase projects (staging + production) set up from scratch
  • Database schemas as versioned setup scripts (tables, indexes, row-level security policies)
  • Supabase Auth: password (immediate post-purchase login, no email wait) plus magic-link as secondary / recovery
  • Entitlements model: a user has zero or more entitlements, each mapped to a Stripe product / price. The app checks entitlements, not raw subscription status
  • Flexible price points map to a single binary "app access" entitlement, so admin can add or change prices in Stripe or GHL anytime with no code changes
  • App access gated by active entitlement. Returning entitled members go straight in
💳
Subscription & payments (Stripe)
  • Stripe recurring subscription billing. Stripe handles billing directly
  • Access gated by active entitlement
  • Standard subscription management: active access, failed-payment handling, cancellation
  • Real-time entitlement sync via Stripe webhook
⚡
CRM sync (GoHighLevel)
  • Real-time, per-Stripe-event webhook sync to GoHighLevel for tagging, segmentation, automation triggers
  • Sync runs in a Supabase Edge Function listening to Stripe webhooks and pushing to GHL on each subscription event
  • Standard Stripe-backed integration confirmed. No native GHL payment handling in scope
🎨
Design & branding
  • Clean, modern, custom UI that feels like the client's brand. Not a generic poker template
  • Custom visual identity for the six player-type avatars is non-negotiable (artwork supplied, integration is in scope)
  • Reference look and flow: the RunOut app, but distinct custom branding
🚀
Deployment & two-layer rollback
  • Deploy the web app to the client's domain via Vercel
  • Two-layer rollback: code via Vercel's one-click deployment revert, content via Supabase versioned publish
  • Two Supabase environments (staging, production) configured for the lifecycle of the product

Forward-Looking Design

Architecture Approach

Decisions absorbed into V1 so future products (Masterclass, member pricing, adaptive AI) become additive, not rebuilds.

🛡

Entitlements over subscription status

A user has zero or more entitlements (quiz subscription, future Masterclass, future products), each mapped to a Stripe product / price. The app checks entitlements, not raw subscription status. Future multi-product support becomes new rows plus a new Stripe product, not a rebuild.

💲

Flexible price points, single access entitlement

Multiple Stripe prices can map to the same binary "app access" entitlement. Admin can add or change price points in Stripe or GHL anytime with no code changes.

🔑

Single Supabase Auth identity from day one

Password login for immediate access post-purchase (no email wait), with magic-link as a secondary / recovery path. Ready for member pricing via Stripe coupons tied to existing entitlements. One identity carries across future products.

⟲

Answer-event logging from day one

Every quiz attempt logged as an event (user, question, correct / incorrect, timestamp, time taken). Powers progress and stats. Also the fuel for adaptive AI recommendations later without a rebuild.

⌥

Stable IDs and rich content tags

Concept, principle, player type, street, difficulty. Tagged from day one so future filtering, search, and AI features have structure to work with.

🧠

pgvector-ready Postgres

pgvector is supported by Supabase Postgres, so vector search and embeddings can be added later as an additive layer. Content text is kept retrievable so this is never blocked. No embeddings generated in this build.

✨

AI-derived data in its own tables

Future AI features stay additive and never force changes to the source content. Authored content remains the single source of truth.

⟲

Two-layer rollback

Code rollback via Vercel's one-click deployment revert. Content rollback via Supabase versioned publish. Code and content can roll back independently without coordinating each other.

▦

Single content pipeline

The validate, stage, publish, rollback pipeline is content-type-agnostic. All V1 content types (Lessons, Tips, References, Path nodes) are authored, versioned, and rolled back through the same machinery. No bespoke per-type pipeline anywhere in V1.

Build Plan

Phase Breakdown (Technical)

Four phases. M2 (the CMS) is the heaviest by design. M4 carries the engagement loop and skills path. Commercial terms are intentionally excluded.

M1
Foundation, Static Table UI, Q&A Interface, Glossary (nested), PWA Scaffold

Visible, tangible product on screen by end of M1. Look and feel anchored, and the technical foundation in place for the heavy CMS work in M2.

  • Custom static 9-max table layout (positions, stacks, dealer button, blinds, pot, board, hole cards), mobile-first
  • Lightweight card-rendering component
  • Multiple-choice question / answer interface (four answers, one correct)
  • Slide-up feedback drawer (correct / incorrect plus teaching commentary)
  • Tap-to-define glossary drawer with nested linking and navigation stack
  • Bulk import flow: paste a JSON batch, validate against the schema, save to staging. Same validator gates as the paid test, productionized into a polished M1 deliverable so the client can ingest large Claude-generated content batches from day one
  • PWA scaffolding: manifest, service worker, caching strategy
  • Entitlements + auth schema design

Nested glossary linking absorbed at no additional scope cost as a refinement of the existing glossary work.

M2
Quiz Engine, CMS Backbone, Today's Tip + Saved Tips + References

The heaviest milestone. The product becomes self-sufficient and the client can author and ship content end-to-end without developer involvement.

  • Quiz / lesson engine: sequencing, scoring, session completion, randomized order, review missed questions
  • Starred / saved questions, persistent across sessions
  • No-code wizard-style authoring UI, branches by question type (general Q&A vs hand scenario)
  • Visual interactive table builder for hand scenarios (seats, stacks, button, hole / board cards)
  • Bulk import / export (CSV, JSON, Markdown) with a clean markdown import spec for use with Claude or other AI tools
  • Zod-based validation engine with field-path-precise, human-readable error messages
  • Staging / production isolation, write-to-staging with preview, versioned publish, one-click content rollback
  • Content type generalization: add a content_type discriminator across the staging, published, and versions tables (rename to content_*) so the validate, stage, publish, rollback pipeline serves Lessons, Tips, References, and Path nodes through the same Edge Functions and the same versioning machinery. Single pipeline, every type
  • Pipeline hardening: server-side re-validation inside the promote and rollback Edge Functions; admin auth gate on the validator UI; CORS allow-list tightened to the actual staging and production Vercel domains
  • Today's Tip surface + Saved Tips list + References Library (cheat sheets, Character Mapping, methodology)
  • Shared scenario data model across wizard, bulk upload, validator, and member app

Starred / saved questions absorbed at no additional scope cost as a refinement of the missed-question review described in the original Loom.

M3
Auth, Entitlements, Stripe Subscriptions, Real-time GHL Sync, Event Logging

Members can subscribe, log in, and have access tied to entitlements. CRM stays in real-time sync. Answer events flow from day one. Standard Stripe-backed GoHighLevel integration confirmed with client. No additional webhook handler needed.

  • Supabase staging + production environments, schemas with row-level security
  • Supabase Auth: password (immediate post-purchase login) plus magic-link recovery
  • Entitlements model live, app gating reads entitlements (not raw subscription state)
  • Flexible price points map to a single binary access entitlement (no code changes for new prices)
  • Stripe recurring subscription billing with webhooks. Failed-payment and cancellation handling
  • Real-time GoHighLevel sync via per-Stripe-event webhook (Supabase Edge Function)
  • Answer-event logging instrumentation: every attempt logged as an event for stats and future AI
M4
Skills Path, Streaks, Push, Badges, Onboarding, Leaderboard, QA, Deploy

Ship it. Engagement loop, guided progression, and the on-phone push experience all in. QA across desktop and mobile, deploy to the client's domain.

  • Skills Path / visual progression (Duolingo-style guided map with skill nodes, unlock states, dependencies)
  • Web push notifications (PWA, streak reminders, VAPID setup, scheduled send job)
  • Badge decay / mastery-maintenance logic
  • Onboarding flow + warm-up lesson + celebratory animations
  • Monthly contest leaderboard: ranked by quiz performance, monthly reset, contest mechanics for community engagement
  • Points / scoring and daily streak
  • Achievement badges for milestones
  • Progress tracking: questions answered, accuracy rate, sessions completed, improvement over time, richer stats dashboard
  • Full QA across desktop and mobile, polish, deployment to client's domain (Vercel)
